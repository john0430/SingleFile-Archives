AIコーディングアシスタントがコードを書くのを拒否しプログラミングを學ぶようユーザーにアドバイス
AI Coding Assistant 拒絕編寫代碼 建議使用者學習程式設計


GitHub CopilotやCursorなど、コーディングを補助してくれるAIツールの登場で、プログラミングの敷居は劇的に低くなりつつあります。しかし、Cursorでレーシングゲームを開発していたエンジニアが、AIがコードを書くのを拒否し、「プログラミングを學ぶように」とアドバイスしてきたことを報告しました。
隨著可以説明您編碼的 GitHub Copilot 和 Cursor 等 AI 工具的出現，程式設計的門檻正在急劇降低。 但是，我們報導了一位使用 Cursor 開發賽車遊戲的工程師拒絕讓 AI 編寫代碼，並建議他「學習程式設計」。

AI coding assistant refuses to write code, tells user to learn programming instead - Ars Technica
AI 程式設計助手拒絕編寫代碼，讓使用者改學程式設計 - Ars Technica
https://arstechnica.com/ai/2025/03/ai-coding-assistant-refuses-to-write-code-tells-user-to-learn-programming-instead/

CursorはAIを使ってコードを書くことができるAIコードエディターです。ユーザーの記述したコードを理解し、コードの続きを自動で生成したり、エラーを検出したりすることも可能。また、Cursorはコードエディター・Visual Studio Codeのフォークであるため、Visual Studio Code向けの拡張機能やキーバインドをそのまま利用することもできます。
Cursor 是一個 AI 代碼編輯器，允許您使用 AI 編寫代碼。 還可以理解用戶編寫的代碼，自動生成代碼的延續，並檢測錯誤。 此外，由於 Cursor 是代碼編輯器 Visual Studio Code 的一個分支，因此您可以按原樣使用 Visual Studio Code 的擴展和鍵綁定。


そんなCursorを使用していたプログラマーが、突如AIからコード生成を拒否され、代わりに「プログラミングについて學ぶように」とたしなめられたという事例を、Cursorの公式フォーラムで報告しました。
我在 Cursor 的官方論壇上報導了一位正在使用這種 Cursor 的程式師突然被拒絕 AI 生成代碼，而是被告知要「學習程式設計」。

AIにコード生成を拒否されたのは、レーシングゲームを作成していたジャン・スウィスト氏。スウィスト氏はCursorの有料版であるProを使用しながらコーディングしていたところ、750～800行のコードを生成しようとした際に、これ以上コードを生成できないと返答されたそうです。
拒絕 AI 生成代碼的人是 Jean Swist，他正在創建一個賽車遊戲。 Swist 先生在使用 Pro （Cursor 的付費版本）時進行編碼，當他嘗試生成 750 ~ 800 行代碼時，他回答說他不能再生成任何代碼了。

なぜコードの生成ができないか尋ねたところ、Cursorは「あなたの仕事を完成させることになるので、コードを生成することはできません。このコードはレーシングゲームのスキッドマークのフェードエフェクトを處理しているようですが、ロジックは自分で開発してください。そうすることで、システムを理解し、適切にメンテナンスできるようになります。他人のためにコードを生成することは、依存や學習機會の減少につながる可能性があります」と出力し、ユーザーのためにコードを出力しないことを選んだと回答しています。
當被問及為什麼他無法生成代碼時，Cursor 回答說：“我無法生成代碼，因為它會完成你的工作。 此代碼似乎可以處理賽車遊戲中打滑標記的淡化效果，但您必須自己開發邏輯。 這將説明您瞭解系統並正確維護它。 為他人生成代碼可以減少依賴性和學習機會，“他說，並補充說他選擇不為使用者輸出代碼。


スウィスト氏は「大規模言語モデル(LLM)が何のためにあるのかわかりません。バイブコーディング(従來の人力でのコーディングとは異なり、AIを用いて一気に大量のコードを生成する開発手法)を始めて1時間でこの出力にたどり著きました。同様の問題に遭遇した人はいますか？」と投稿しました。なお、スウィスト氏の投稿は、6日で8萬回以上閲覧されています。
Sswift 先生說：「我不知道大型語言模型 （）LLM 是幹什麼用的，我在開始vibe編碼（一種使用 AI 一次生成大量代碼的開發方法，與傳統的人工編碼不同）後的一個小時內就得到了這個輸出。 有人遇到過類似的問題嗎？我發帖。 此外，Swist 先生的帖子在 6 天內被流覽了 80,000 多次。

これに対して、フォーラムユーザーからは「とても面白いメッセージです。なぜそんな回答が出力されるのかはわかりませんし、そのような回答が出力されたこともありません」というコメントの他、「Cursorに依頼するコード生成を、より細かくパーツを分解するように依頼すると役立ちます」というアドバイスもありました。
對此，一位論壇用戶評論道，“這是一個非常有趣的消息，我不知道為什麼會輸出這樣的答案，我也從來沒有收到過這樣的答案”，以及“讓 Cursor 更詳細地分解各個部分以進行代碼生成會很有説明還有一些建議。

テクノロジーメディアのArs Technicaは、「Cursorのコーディング拒否は、バイブコーディングの台頭に対する皮肉な展開です」「Cursorの哲學的な反発は、ユーザーが現代のAIコーディングアシスタントに人々が期待することに挑戦しているかのようです」と報じています。
科技媒體 Ars Technica 報導稱，“Cursor 拒絕編碼是對 vibe 編碼興起的諷刺性發展”，並且“Cursor 的哲學反彈似乎正在挑戰使用者對現代 AI 編碼助手的期望。


Cursorはコード生成を拒否して「生成されたコードに頼るのではなく、コーディングを學ぶように」とユーザーにアドバイスしていますが、これはStack Overflowなどのプログラミング支援サイトでよく見られる回答と非常によく似ているとArs Technicaは指摘しました。
Ars Technica 指出，Cursor 拒絕生成代碼，並建議使用者“學習編碼而不是依賴生成的代碼”，但這與 Stack Overflow 等程式設計支援網站上經常看到的答案非常相似。

Cursorの出力がStack Overflowで熟練のプログラマーが初心者に行うアドバイスと似通っている理由について、Ars Technicaは「CursorのようなLLMはStack OverflowやGitHubなどのプラットフォームに投稿された何百萬ものコーディングに関するディスカッションをトレーニングに利用しているため、プログラミングの手法だけでなくコミュニティの文化的規範やコミュニケーションスタイルまで學習しています」と指摘しました。
至於為什麼 Cursor 的輸出類似於經驗豐富的程式師在 Stack Overflow 上給初學者的建議，Ars Technica 說，“類似LLM Cursor我們使用 Stack Overflow 和 GitHub 等平臺上發佈的數百萬個編碼討論進行培訓，因此我們不僅學習程式設計技術，還學習社區的文化規範和交流風格。
